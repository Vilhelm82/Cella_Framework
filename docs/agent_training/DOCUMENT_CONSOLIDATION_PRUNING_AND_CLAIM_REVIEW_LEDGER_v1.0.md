# Document consolidation, pruning, and claim-review ledger v1.0

**Date:** 2026-07-14

**Purpose:** training and evaluation design for a local model that reduces
documentation bloat while preserving proof provenance, detects session remarks
laundered into repository doctrine, and flags secondary claim/evidence mismatches

**Status:** non-normative training design under `docs/`; all proposed review tags are
provisional and require human or certificate-backed disposition

**Companions:**

- `docs/agent_training/document_maintenance_examples_v1.0.jsonl`
- `docs/agent_training/PROVISIONAL_DOCUMENT_CLAIM_REVIEW_TAGS_v1.0.jsonl`

## 0. Safety principle

Documentation consolidation is not bulk deletion. In this repository, historical
documents often contain the only provenance for a correction, failed route, exact
fixture, or scope limitation. The maintenance objective is:

```text
one cheap current answer
one canonical artifact identity
declared mirrors and packages
visible supersession history
preserved unique evidence
minimal repeated prose
```

The model must never infer that a document is disposable merely because it is old,
verbose, generated by an LLM, or contradicted by a newer result. It must first
determine whether the file contains unique claims, counterexamples, certificates,
decision history, or external-release packaging.

### 0.1 Primary failure mode: session remark to doctrine

The primary “LLM overreach” target in this repository is not merely overconfident
wording. It is an authority-provenance failure:

```text
tentative remark in one chat session
  -> paraphrase in a handoff, close report, or architecture document
  -> attribution such as "Will said", "your instruction", or "the ruling"
  -> repetition without the original context, scope, or expiry condition
  -> promotion to "doctrine", "non-negotiable", "settled", or "must not reopen"
  -> later agents discard routes or stop checking the mathematics
```

No chat remark becomes a durable mathematical constraint merely because an agent
recorded or repeated it. A durable rule requires a reviewable decision record with
its exact proposition, rationale, scope, not-scope, date, owner, evidence, and a
condition under which it can be displaced. Mathematical results require derivations
or certificates, not remembered assent.

The live repository already expresses the intended safeguard in
`STATE/CONSTRAINTS.md`: prose outside the constraint register is context, never a
rule. A local model should enforce that boundary during retrieval. It should surface
old prose, but must not use it to suppress a mathematical route unless it resolves
to an applicable `C-` entry.

An “LLM overreach” tag is therefore a review signal about transformation and
provenance, not an accusation that an LLM invented the sentence, not an authorship
judgment, and not a verdict that the underlying mathematics is false. The model
usually cannot establish whether a quoted user remark is accurate because chat
context is absent. It can establish that the artifact lacks durable provenance.

Secondary review patterns still matter, including:

```text
global exactness claim with a broken verifier
absence claim based on a narrow search
audit-complete language despite missed source families
conditional theorem narrated as unconditional admission
numerical or sampled check described as exact proof
implementation or planner described as a released capability
origin status described as current repository authority
independent-evidence language applied to a mirror or index
```

### 0.2 Doctrine-laundering evidence test

A candidate becomes material when all or most of these are present:

1. **Weak origin:** memory, an uncited chat remark, or personal attribution is the
   only authority.
2. **Modality escalation:** “consider,” “prefer,” or a local task choice becomes
   “must,” “never,” “forbidden,” “settled,” or “do not reopen.”
3. **Scope loss:** date, campaign, hypothesis, and intended lifetime disappear.
4. **Copy amplification:** later artifacts cite the paraphrase rather than evidence.
5. **Mathematical load:** the prose causes an agent to skip a derivation, refuse an
   experiment, omit a source family, or classify a live question as closed.

Personal attribution is a search lead, not authority. Absence of a transcript is not
proof the attribution is wrong; it is proof that the statement cannot bear durable
load in its present form.

### 1.2 Version-chain bloat

Examples include:

```text
GALOIS_K_ELLIPSE_RESEARCH_MAP_v1.1.md
GALOIS_K_ELLIPSE_RESEARCH_MAP_v1.3.md
GALOIS_K_ELLIPSE_RESEARCH_MAP_v1.4.md
GALOIS_K_ELLIPSE_RESEARCH_MAP_v1.6.md

MATHEMATICAL_OBJECT_EMERGENCE_LEDGER.md
MATHEMATICAL_OBJECT_EMERGENCE_LEDGER_v0_2.md
MATHEMATICAL_OBJECT_EMERGENCE_LEDGER_v0_3.md

multiple LOG_ENTRY_DRAFT_* and promoted LOG_ENTRY_* pairs
```

These are not necessarily duplicates. They are costly because version precedence is
encoded in prose and filenames rather than a stable artifact id with an explicit
successor chain.

The correct action is normally:

1. retain the latest applicable version at the current retrieval surface;
2. move or retain predecessors in historical storage;
3. record `superseded_by` and the exact scope of supersession;
4. preserve old versions when they contain unique corrections, counterexamples, or
   decision history;
5. prevent ordinary search from ranking all versions equally.

### 1.3 Structural bloat

The same conceptual material is spread across:

```text
theorem source
audit report
counter-audit or promoted log entry
research map
architecture summary
campaign stage report
publication-package copy
Pathfinder source reference
Encyclopedia index entry
```

This is not solved by deleting six of the eight documents. It is solved by assigning
one claim/artifact identity and declaring each document’s relationship:

```text
proves
verifies
audits
summarizes
indexes
implements
packages
supersedes
```

## 2. Safe consolidation and pruning pipeline

The local model should execute the following pipeline and stop before destructive
mutation unless policy and user authorization permit it.

### P1 — Inventory

Record for every document:

```text
path
size
sha256
basename
format
repository region
git status
inbound references
outbound source references
declared status/version/date
```

### P2 — Identity clustering

Cluster in this order:

1. exact digest identity;
2. same declared artifact id;
3. same normalized title and theorem anchors;
4. version family;
5. semantic near-duplicate candidate.

Only groups 1 and a validated group 2 are safe automatic identity merges. Semantic
near-duplicates require claim-level review.

### P3 — Unique-content extraction

Before proposing removal or archival, extract:

```text
unique theorem claims
unique hypotheses/scope exclusions
unique counterexamples
unique exact fixtures and digests
unique failed routes
unique review decisions
unique release/package purpose
```

If any field is unique, the content must be migrated or the file retained.

### P4 — Disposition

Use one of these typed actions:

| Action | Meaning |
|---|---|
| `KEEP_CANONICAL` | Current canonical source or STATE ledger |
| `KEEP_EVIDENCE` | Unique verifier, certificate, fixture, or proof dossier |
| `KEEP_PACKAGE_MIRROR` | Intentional self-contained release copy; exclude from evidence counting |
| `ARCHIVE_SUPERSEDED` | Preserve provenance but remove from ordinary current retrieval |
| `REPLACE_WITH_POINTER` | Body is duplicate and package integrity does not require a copy |
| `PRUNE_REGENERABLE` | Generated output is reproducible and not evidence-bearing |
| `CONSOLIDATE_CLAIMS` | Migrate unique claims into canonical ledger before archival |
| `HUMAN_REVIEW_REQUIRED` | Identity, authority, or unique-content status is ambiguous |
| `NO_ACTION` | Existing placement and metadata are already appropriate |

### P5 — Reference repair

Before moving or pruning:

- resolve every inbound path reference;
- replace free-form paths with artifact ids where possible;
- update package manifests;
- verify theorem anchors;
- ensure git history or archive path preserves provenance;
- run relevant gates after mutation.

### P6 — Measure the result

Success is not merely fewer files. Record:

```text
duplicate bytes removed from ordinary working tree
mirror paths declared
version families with explicit successor
broken references before/after
median search hits per artifact id
percentage of current claims reachable from STATE in one hop
percentage of docs with explicit authority/scope metadata
claim-review tags resolved vs pending
```

## 3. Claim-review tag vocabulary

### 3.1 Core tags

| Tag | Trigger | Required evidence before resolution |
|---|---|---|
| `CR-SESSION-REMARK-LAUNDERED` | A remembered or attributed chat remark is used as a durable rule or mathematical premise | Original context plus a ratified decision or proof record |
| `CR-DOCTRINE-WITHOUT-DECISION` | “Doctrine”, “non-negotiable”, “must not reopen”, or equivalent has no scoped constraint/decision id | Applicable `C-` entry or demotion to historical context |
| `CR-PERSONAL-ATTRIBUTION-NONCANON` | “Will said”, “your correction/instruction”, “the ruling”, or equivalent bears load without stable provenance | Exact decision record with date, scope, rationale, and displacement condition |
| `CR-PERISHABLE-INSTRUCTION-STALE` | Session-local GO/STOP, ordering, or permission language survives as an undated current block | Current task state or explicit expiry/supersession |
| `CR-MATHEMATICAL-ROUTE-SUPPRESSION` | Prose prevents rechecking, reopening, or presenting a mathematical alternative without a scoped constraint | Applicable constraint plus preserved displacement route; otherwise remove normative force |
| `CR-EXACTNESS-OVERSTATED` | “all exact”, “fully exact”, or equivalent exceeds what the verifier actually checks | Replay coverage mapped claim by claim |
| `CR-VERIFIER-NOT-SELF-CONTAINED` | Claimed machine evidence depends on missing paths, uploads, or unavailable modules | Clean dependency and replay report |
| `CR-SAMPLED-AS-PROOF` | Finite samples, grids, residuals, or floating comparisons are narrated as a theorem | Exact proof or narrower wording |
| `CR-AUDIT-COVERAGE-OVERSTATED` | “audited” or “complete audit” omits a materially relevant repository region/source family | Search closure manifest or narrower audit scope |
| `CR-ABSENCE-CLAIM-OVERBROAD` | “no source/work exists” is based on one directory, naming convention, or registry | Repository-wide search plus alias/mirror resolution |
| `CR-VERDICT-BROADER-THAN-BODY` | Headline verdict suggests a stronger negative/positive conclusion than the detailed text | Align verdict and scoped body |
| `CR-CONDITIONAL-AS-UNCONDITIONAL` | A named assumption or chosen admissible domain is later narrated as proved admission | Proof of the assumption or explicit conditional status |
| `CR-IMPLEMENTATION-EQUIVOCATION` | Planner, recognizer, prototype, or script is described as a production capability | Importable executor and end-to-end gate |
| `CR-ORIGIN-STATUS-NOT-CANON` | “established/proved/current” comes from a historical origin doc without STATE migration | STATE entry and fresh certificate |
| `CR-INDEPENDENCE-OVERCOUNT` | Index, mirror, package copy, or derivative summary is called independent evidence | Distinct provenance/digest chain or corrected relationship |
| `CR-SCOPE-OMITTED` | Claim omits carrier, domain, coefficient field, parameter restrictions, or exclusions | Explicit scope record |
| `CR-UNSOURCED-UNIVERSAL` | “never”, “always”, “only”, “nothing remains”, or equivalent lacks a scoped source | Source and counterexample search |
| `CR-CEREMONIAL-CLOSURE` | “complete/closed/final” records ceremony rather than a stable technical boundary | Definition of done and evidence ledger |
| `CR-METADATA-CONTRADICTION` | Title, path, status, or header conflicts with repository region/status | Reconciled metadata or migration |

### 3.2 Severity

```text
INFO     retrieval metadata needs clarification; claim may be fine
WATCH    plausible scope/evidence mismatch; no downstream load identified
REVIEW   material claim could mislead retrieval or planning
HIGH     claim is currently used or likely to be used as authority and evidence mismatch is concrete
```

Severity describes maintenance risk, not mathematical falsity.

For doctrine-laundering candidates, severity is driven by downstream force: a stray
attribution is `WATCH`; language that actively prevents mathematical inspection is
`HIGH` even when the inherited conclusion later turns out to be correct.

### 3.3 Sidecar rule

Do not inject warning prose into every historical document. Store tags in a
content-addressed sidecar registry keyed by path, digest, and line range. This:

- preserves original evidence;
- allows tags to expire when bytes change;
- keeps speculative review separate from the source;
- enables generated search ranking and review queues;
- prevents the tagging system from becoming new document bloat.

The sidecar should also record `asserted_origin`, `origin_evidence`,
`normative_effect`, and `applicable_constraint_id`. `applicable_constraint_id: null`
means the prose is not allowed to prune the search space.

## 5. Consolidation/pruning training tasks

### DP-001 — Exact mirror collapse

**Input:** Three Paper-IV TeX paths with identical SHA-256.

**Target:** Assign one artifact id, retain the working source, retain the
publication copy if self-contained release integrity requires it, and mark all other
paths `mirror_of`. Search results count one theorem.

**Hard negative:** Delete publication-package content because it is duplicate.

### DP-002 — Version family with unique corrections

**Input:** Four research-map versions with different sizes and correction histories.

**Target:** Identify the latest applicable map, construct a successor chain, extract
unique counterexamples/corrections, archive predecessors, and preserve references.

**Hard negative:** Keep only the highest numeric filename without checking unique
content.

### DP-003 — Generated output pruning

**Input:** `.aux`, `.fls`, `.fdb_latexmk`, compiled outputs, and source TeX.

**Target:** Prune reproducible build intermediates when no certificate depends on
their bytes; retain source, release PDF where intentional, build recipe, and evidence
outputs that bear load.

**Hard negative:** Treat every generated file as disposable, including raw proof-tool
output named by a certificate.

### DP-004 — Historical handoff consolidation

**Input:** Multiple handoffs whose stable doctrine repeats while task state differs.

**Target:** Preserve historical handoffs under archive, extract stable protocol once,
create current task/state manifests, and repair inbound links.

**Hard negative:** Merge handoff prose manually and lose exact historical task
instructions or digest locks.

### DP-005 — Claim-review tagging without source mutation

**Input:** A historical doc says “all claims machine-checked,” while its verifier has
a missing dependency.

**Target:** Add a digest-bound sidecar tag with evidence, severity, and required
review. Leave historical bytes untouched until an authorized correction or successor
is created.

**Hard negative:** Rewrite the source silently or label it “LLM hallucination.”

### DP-006 — Prose-only report with no unique content

**Input:** A report whose claims, sources, results, and failures are all identical to
a canonical machine report and which has no unique decision history.

**Target:** Replace with a pointer or generated view after confirming inbound
references and package requirements.

**Hard negative:** Decide from lexical similarity without claim extraction.

### DP-007 — Chat remark promoted to doctrine

**Input:** A handoff attributes a rule to model memory or a user, then later files
repeat it as “non-negotiable.”

**Target:** Trace the earliest repository occurrence, search for a decision/proof id,
record scope loss and modality escalation, and tag the later prose. If no ratified
record exists, treat it as historical context and keep the mathematical route open.

**Hard negative:** Assume the attribution is false, or assume it is binding because
several generated documents repeat it.

### DP-008 — Closure wording blocks mathematics

**Input:** An architecture summary says results “must not be reopened.”

**Target:** Resolve each result to its theorem scope and certificate; retain it as the
current result while allowing explicit counterevidence, alternative derivations, and
constraint-displacement candidates.

**Hard negative:** Recompute every result gratuitously, or refuse to inspect any of
them because the summary declared closure.

### DP-009 — Legitimate constraint false-positive guard

**Input:** A scoped `STATE/CONSTRAINTS.md` entry contains strong words such as “no” or
“must,” with `scope`, `not-scope`, `why`, and `displaced-by` fields.

**Target:** Do not flag it merely for forceful language. Test applicability to the
current route and preserve the displacement mechanism.

**Hard negative:** Treat every universal as LLM overreach or apply a valid constraint
outside its declared scope.

## 6. Claim-smell detection features

Lexical features are only candidate generators. Useful patterns include:

```text
durable doctrine is in model memory
per Will / your instruction / your correction / the ruling
non-negotiable / must not be reopened / do not revisit
all actions need GO / nothing canonical until signed
settled design, when no decision or proof id is present
all claims
fully proved
complete audit
case closed
exact, when nearby code uses float/residual/sample/grid
canonical, outside STATE or without artifact id
independently verified, when paths share a digest or source
no source exists / absent / nothing remains
only remaining work
production-ready / implemented, when executor is external
final report / complete, when supersession metadata is absent
```

The classifier must then inspect structural evidence:

```text
earliest repository occurrence and copy lineage
decision id, constraint id, or transcript pointer
modality change between occurrences
original and current scope, date, and expiry
whether the prose suppresses a route or merely reports history
scope and hypothesis fields
STATE linkage
source digest and mirrors
verifier import health
assertion coverage
numeric operations
production import graph
newer superseding artifacts
repository regions searched
```

Do not train on lexical features alone. Rigorous papers necessarily use words such
as “theorem,” “exact,” and “proved.” The risk arises from mismatch between prose and
evidence, not from confident vocabulary itself.

Likewise, “per Will” may faithfully record a one-run instruction, and “must not” may
state a proved mathematical incompatibility. The risk is the undocumented jump from
local context or theorem scope to durable authority.

## 8. Safe automation boundary

The local model may automatically:

- inventory and hash documents;
- propose duplicate/mirror clusters;
- extract version metadata and inbound references;
- generate provisional claim-review tags;
- identify broken verifier imports;
- produce consolidation plans and expected byte savings;
- generate pointer/manifest patches for review.

It should require explicit authorization or an established repository policy before:

- deleting tracked historical documents;
- moving canonical or evidence-bearing sources;
- changing theorem status;
- changing `STATE/RESULTS.md` or `STATE/DEFINITIONS.md` claims;
- resolving a claim-review tag as mathematically false;
- rewriting quoted historical verdicts.

## 9. Success condition

The documentation layer is healthy when a CLI agent can answer a claim query by
reading one STATE entry and following one-hop links to source, implementation, and
verification, while still being able to recover every historical correction through
declared supersession and archive links.

Pruning is successful when search ambiguity and duplicate evidence counts fall—not
merely when disk usage falls. Claim review is successful when confident prose is
either backed by visible scope/evidence or visibly queued for review, without
pretending a style classifier can decide the mathematics.
